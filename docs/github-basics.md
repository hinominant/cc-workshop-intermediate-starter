# Git / GitHub 入門ガイド

このワークショップで必要な Git の基礎知識をまとめています。

---

## Git とは

ファイルの変更履歴を記録・管理するツール。「いつ・誰が・何を変えたか」を追跡できる。
GitHub はその履歴をクラウドに保管し、チームで共有するためのサービス。

---

## 必須コマンド

### リポジトリのコピーを手元に取得
```bash
git clone https://github.com/<username>/<repo>.git
cd <repo>
```

### 現在の状態を確認
```bash
git status
```
- 赤色: 変更はあるが未追跡（ステージングされていない）
- 緑色: ステージング済み（次のコミットに含まれる）

### 変更の差分を確認
```bash
git diff                  # ステージング前の差分
git diff --staged         # ステージング済みの差分
```

### 変更をステージング（コミット候補に追加）
```bash
git add src/security/signature.ts    # ファイル指定
git add src/security/                # ディレクトリ指定
```

> `.env` や秘密情報を含むファイルは絶対に `git add` しない

### コミット（変更を確定）
```bash
git commit -m "feat: HMAC-SHA256 署名検証を実装"
```

### 履歴を確認
```bash
git log --oneline -10     # 直近10件を1行表示
```

## よくある失敗と対処法

### 1. 秘密情報をコミットしてしまった

**最も危険なミス。** Git の履歴に残るため、後からファイルを消しても復元できる。

```bash
# 直前のコミットに .env が含まれていた場合
git reset HEAD~1          # コミットを取り消す（ファイルは残る）
# .gitignore に .env が入っていることを確認してから再コミット
```

> 既に push した場合は、**秘密情報を即座にローテーション**（新しい値に変更）する

### 2. "nothing to commit" と言われた

```bash
git status                # 変更を確認
git add <ファイル>         # ステージングを忘れていないか確認
```

### 3. コンフリクト（衝突）が発生した

同じファイルの同じ場所を複数人が編集した場合に起きる:

```
<<<<<<< HEAD
自分の変更
=======
相手の変更
>>>>>>> branch-name
```

対処: `<<<<<<<` `=======` `>>>>>>>` の行を削除し、正しい内容に編集してからコミット。

---

## Tips

- **こまめにコミット**: 1つの機能が動いたらコミット。大きな変更を1回でコミットしない
- **`.env` はコミットしない**: `.gitignore` に含まれていることを確認
- **コミットメッセージは具体的に**: `fix: バグ修正` ❌ → `fix: タイムスタンプ検証の境界値バグを修正` ✅
- **変更前に `git status`**: 今の状態を把握してから作業する
